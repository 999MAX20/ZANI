# Zani UI/UX Reference Analysis

Дата: 15.05.2026

Источник: `/Users/maksim/Desktop/Zani/references`

Цель документа: превратить собранные UI/UX-референсы в практические правила для будущего редизайна Zani. Это не инструкция копировать чужие экраны. Это продуктовая карта: какие паттерны брать, какие адаптировать под SMB CRM, а какие не переносить.

## 1. Главный вывод

Zani должен выглядеть не как тяжелая CRM и не как декоративный AI dashboard. Лучшее направление по референсам:

- спокойный premium SaaS;
- высокая читаемость;
- крупные, уверенные заголовки;
- светлый фон;
- белые рабочие поверхности;
- мягкие границы и тени;
- минимум визуального шума;
- быстрые действия рядом с данными;
- AI как помощник внутри процесса, а не отдельный спектакль.

Самый сильный визуальный ориентир для Zani: сочетание чистоты MoonAI, коммуникационной логики Intercom, карточки сделки уровня amoCRM/Bitrix24 и мобильной простоты современных task/CRM-приложений.

## 2. Стратегия: что делать сейчас

Не стоит сейчас полностью переписывать техплан только под 10 000 активных пользователей. Правильнее:

1. Доделать актуальные продуктовые техпланы по текущим этапам.
2. После каждого крупного блока делать audit: что реализовано, что работает, что требует hardening.
3. На основе audit собрать отдельный production-readiness план под 10 000 merchants.
4. Уже сейчас держать архитектурные guardrails под масштаб:
   - tenant isolation;
   - индексы на бизнес-критичных таблицах;
   - audit log;
   - object-level permissions;
   - rate limiting;
   - async jobs для AI/уведомлений/интеграций;
   - понятная observability-стратегия;
   - запрет на синхронные тяжелые AI/analytics операции в user request.

Причина: если уйти в масштабирование до завершения ключевых CRM-флоу, мы начнем оптимизировать еще неустоявшийся продукт. Но если не держать правила масштабирования уже сейчас, потом придется дорого переделывать ядро.

## 3. Dashboard

Референсы: `dashboard/*`, особенно MoonAI-подобные экраны.

Что работает:

- крупный page title;
- период/фильтры рядом с заголовком;
- 3-5 KPI cards в первом экране;
- спокойные карточки без лишних градиентов;
- графики только там, где они помогают принять решение;
- навигация от метрики к действию.

Как применить в Zani:

- Dashboard должен отвечать на вопросы: что случилось, что горит, что сделать сейчас.
- Главный блок должен быть не “аналитика ради аналитики”, а “Что требует внимания”.
- KPI: новые заявки, просроченные задачи, записи сегодня, конверсия, выручка/потенциальная сумма.
- AI-блок оставить компактным: 2-3 рекомендации, каждая с действием.
- Убрать тяжелые графики, кривые прогресса и декоративные AI-инсайты, если они не ведут к действию.

Не делать:

- перегружать первый экран BI-виджетами;
- делать “космический” AI-дизайн;
- показывать больше 5-7 равнозначных карточек;
- использовать таблицы как основной формат dashboard.

## 4. Sidebar и навигация

Референсы: MoonAI agent screens, dashboard screens.

Что работает:

- сильная левая навигация с группами;
- вложенные разделы для сложных зон;
- активный пункт хорошо заметен;
- разделы по смыслу, а не по внутренней архитектуре;
- onboarding/first launch как отдельная ненавязчивая зона.

Как применить в Zani:

- Навигация должна быть сгруппирована:
  - Рабочая зона: Dashboard, Inbox, Leads, Deals, Tasks, Calendar.
  - Клиенты: Clients, Segments, Timeline.
  - Рост: Automations, Campaigns, Landing, Integrations.
  - Управление: Analytics, Team, Roles, Settings.
- На mobile sidebar должен быть читаемым: контраст текста, нормальные touch targets, закрытие после выбора пункта.
- Не прятать ежедневные разделы глубоко.

Не делать:

- темный полупрозрачный sidebar с плохим контрастом;
- навигацию по техническим названиям сущностей;
- слишком длинный список без групп.

## 5. Unified Inbox / Conversations

Референсы: `inbox/Unified Inbox.png`, Intercom-подобная логика.

Что работает:

- трехзонная структура: список диалогов, активный чат, CRM-контекст;
- поиск и фильтры над списком;
- быстрые статусы: все, ошибки, пауза, handoff, новые;
- пустое состояние “Выберите диалог”;
- связанные сущности должны быть рядом с перепиской.

Как применить в Zani:

- Inbox должен быть triage-центром, а не просто списком сообщений.
- В карточке диалога показывать: клиент, канал, последнее сообщение, SLA/ожидание, ответственный, связанный lead/deal.
- В правом контексте: клиент, активная заявка/сделка, задачи, заметки, quick actions.
- AI-подсказки должны вставляться как черновик ответа, а не открываться отдельным “AI-театром”.

Не делать:

- отдельный popup для AI-ответа, если менеджер пишет в чате;
- технические raw-id как главный текст;
- скрывать CRM-контекст от менеджера.

## 6. Kanban / Pipeline

Референсы: `kanban/*`.

Что работает:

- крупные цветные stage columns;
- stage header с количеством и суммой;
- карточка показывает только essentials;
- понятные priority/status chips;
- drag-and-drop должен иметь очевидный feedback.

Как применить в Zani:

- Kanban должен быть главным рабочим экраном продаж/заявок.
- Карточка: клиент, услуга/сделка, сумма, источник, SLA, next action, ответственный.
- Потерянную сделку нельзя просто скрыть: нужен reason, audit event, восстановление из архива.
- Для SMB нужен простой режим по умолчанию и расширенный режим для компаний 20-100+ сотрудников.

Не делать:

- вставлять в card много кнопок;
- показывать все поля CRM прямо в kanban;
- позволять удалять лиды/сделки без audit trail.

## 7. Calendar / Scheduling

Референсы: `calendar/*`.

Что работает:

- календарь как полноценная рабочая поверхность;
- сотрудники/ресурсы в колонках;
- время слева;
- colored appointments;
- карточка записи с клиентом, услугой, оплатой, тегами;
- быстрый выбор сотрудника/ресурса.

Как применить в Zani:

- Для салонов/клиник/сервисов календарь должен быть не вторичным экраном, а одним из главных модулей.
- Нужны режимы: день, неделя, список.
- На desktop: сетка времени + ресурсы.
- На mobile: список записей дня, компактный date picker, быстрый CTA “Создать запись”.
- Запись должна открываться в drawer с клиентом, lead/deal, статусом, оплатой, уведомлениями.

Не делать:

- огромные пустые слоты без полезных действий;
- мелкий date picker;
- сложный enterprise calendar на первом экране mobile.

## 8. CRM Card / Client / Deal Detail

Референсы: `crm-card/*`.

Что работает:

- карточка сделки как главный центр работы;
- сверху pipeline stages;
- слева финансовые и клиентские данные;
- справа timeline, дела, комментарии, сообщения, задачи;
- быстрые действия: звонок, email, WhatsApp, документ, счет.

Как применить в Zani:

- Lead/Deal/Client detail лучше делать как drawer или full-page workspace.
- Верх: identity, статус, ответственный, pipeline stage, сумма/ценность, quick actions.
- Табы: Overview, Timeline, Tasks, Messages, Notes, Files.
- Все важные действия должны оставлять activity event.
- Для руководителя видны audit/security элементы, для менеджера только рабочие действия.

Не делать:

- 10+ табов в первом MVP;
- всю карточку как одну длинную форму;
- редактирование без истории изменений.

## 9. Mobile UX

Референсы: `mobile/*`.

Что работает:

- bottom navigation;
- большие touch targets;
- карточки вместо таблиц;
- full-screen drawers;
- thumb-friendly primary action;
- воздух между элементами.

Как применить в Zani:

- Mobile должен быть не “уменьшенной desktop CRM”, а отдельным рабочим режимом.
- Таблицы: заменить card list.
- Фильтры: bottom sheet.
- Detail drawer: full-screen.
- Header: минимум текста, поиск и уведомления и профиль.
- Основные mobile разделы: Главная, Заявки, Сделки, Календарь, Чат.

Не делать:

- горизонтальный скролл таблиц;
- длинные поисковые строки в header на mobile;
- мелкие icon-only кнопки без понятной зоны нажатия.

## 10. AI Agent Settings

Референсы: `agent_settings_screens/*`.

Что работает:

- AI-агент представлен как отдельная сущность с настройками;
- настройки разделены на понятные вкладки:
  - Настройки;
  - Промптинг;
  - Сообщения;
  - LLM-модели;
  - Контроль;
  - Функции;
  - База знаний;
  - Интеграции;
  - Каналы.
- большие формы с четкими блоками;
- toggles для включения/выключения возможностей;
- test chat как безопасная песочница.

Как применить в Zani:

- AI не должен быть везде одинаковой “магической кнопкой”.
- Для бизнеса нужен экран “AI-сотрудники”:
  - роль агента;
  - каналы;
  - база знаний;
  - разрешенные действия;
  - handoff правила;
  - лимиты и контроль;
  - журнал ошибок.
- Начинать с простого режима, advanced настройки раскрывать постепенно.

Не делать:

- давать AI возможность менять CRM-данные без audit;
- показывать пользователю технические параметры моделей как основной сценарий;
- смешивать AI настройки с общими настройками бизнеса.

## 11. Roles / Permissions UX

Референсы в папке `settings-roles` отсутствуют, но логика вытекает из CRM-card, agent settings и enterprise CRM анализа.

Как проектировать:

- Для SMB по умолчанию нужны role presets:
  - Владелец;
  - Администратор бизнеса;
  - Руководитель продаж;
  - Менеджер;
  - Оператор чата;
  - Маркетолог;
  - Бухгалтер;
  - Наблюдатель.
- Расширенная матрица permissions должна быть спрятана под “Расширенные настройки”.
- Опасные действия: удаление, экспорт, массовые изменения, смена ответственного, просмотр финансов, просмотр audit log.
- Все удаления должны быть soft delete + audit + restore.

UX-правило: владелец должен думать не “какой permission включить”, а “что этот сотрудник должен уметь делать”.

## 12. Visual System

Рекомендуемые принципы:

- Светлый фон: `#F6F8FB` / `#F8FAFC`.
- Карточки: белые или почти белые.
- Текст: темный navy/graphite, не чистый черный везде.
- Primary accent: синий/индиго.
- Secondary accent: бирюзовый/зеленый для success.
- Warning: amber.
- Error: soft red.
- Border: тонкий `1px`, низкий контраст.
- Radius:
  - buttons: 12-16px;
  - cards: 16-24px;
  - drawers/modals: 20-28px.
- Shadows: мягкие, только для floating surfaces.
- Иконки: lucide, не смешивать несколько стилей.

Важно: текущая CRM не должна быть залита градиентами. Градиенты уместны в CTA, AI-badge, onboarding, но не как фон каждого блока.

## 13. Component Rules

Header:

- Desktop: поиск, business switcher, language selector, notifications, profile/logout.
- Mobile: menu, search icon, notifications, profile. Без длинной строки и без длинного названия бизнеса.

Search:

- Запрос вводится прямо в header input на desktop.
- На mobile search открывается full-screen или bottom sheet.
- Результаты: grouped by Clients, Leads, Deals, Appointments, Messages, Tasks.
- Popup должен закрываться по Escape, outside click, выбору результата, смене маршрута.

Buttons:

- Primary action один на экран.
- Secondary actions менее заметные.
- Icon buttons минимум 40x40 desktop, 44x44 mobile.
- Иконки уведомлений и закрытия модалок должны быть визуально равны остальным action icons.

Tables:

- Desktop: таблица допустима для списков.
- Mobile: карточки.
- Empty state должен объяснять следующее действие.

Drawers:

- Desktop: right drawer.
- Mobile: full-screen.
- Close button минимум 40x40, иконка 20-24px.

## 14. Что улучшить в текущем UI Zani

Высокий приоритет:

- Укрепить sidebar: контраст, читаемость, закрытие после выбора на mobile.
- Привести header к desktop/mobile паттернам.
- Сделать глобальный поиск настоящим workflow-инструментом.
- Перевести таблицы на mobile card lists.
- Упростить dashboard: меньше декоративных AI/analytics blocks, больше action blocks.
- Calendar довести до полноценного scheduling surface.
- Добавить нормальные detail drawers для lead/client/deal/appointment.

Средний приоритет:

- Единые skeleton/loading states.
- Единая система status chips.
- Единые empty states.
- Настройки AI-агентов в стиле agent settings references.
- Коммуникационный центр как Intercom-like triage.

Низкий приоритет:

- Темная тема.
- Большие графики.
- Расширенные кастомизации внешнего вида.
- Сложные анимации.

## 15. Как не потерять простоту SMB

Главное правило: мощность через progressive disclosure.

- Малому бизнесу с 1-5 людьми показываем простой режим.
- Командам 20-100+ людей даем роли, отделы, audit, SLA, automations, но не на первом экране.
- Каждая сложная настройка должна иметь preset.
- Каждая автоматизация должна иметь шаблон.
- Каждый AI-инсайт должен иметь действие.
- Каждый экран должен иметь понятный “следующий шаг”.

Zani не должен конкурировать с Bitrix24 количеством видимых кнопок. Он должен конкурировать результатом: быстрее принять заявку, не потерять клиента, вовремя ответить, видеть ответственность сотрудников.
